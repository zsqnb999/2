## 一、会直接导致“跑不出正确流”的硬错误

### 1) 车 -> 服务位边把 `cap` 和 `cost` 写反了

代码（建图三重循环里）：

```cpp
ps[i].push_back({n+(j-1)*n+k, t[i][j]*k, 1, ...});
```

`node` 定义是：

```cpp
struct node{ ll v,c,w,ds; };
```

所以这行实际含义是：

- `c = t[i][j] * k`
- `w = 1`

这是错的。正确应是：

- `c = 1`（一辆车只能占一个位置）
- `w = t[i][j] * k`（费用）

**后果**：目标函数被改成“尽量少走边 / 常数边权”，不是最小等待时间；容量语义也坏掉。

---

### 2) 源点到车辆边容量写成 0，流根本出不来

代码：

```cpp
ps[S].push_back({i,0,1,ps[i].size()});
```

这里 `c=0`，意味着 **S->i 没有可用容量**。

同样，服务位到汇点也写成了 `c=0`：

```cpp
ps[i+n].push_back({T,0,1,ps[T].size()});
```

**后果**：从源点无法向外发送任何正流，算法要么直接找不到可增广路，要么行为异常；结果不可能是题目所需最优值。

## 二、会导致越界 / 未定义行为的错误

### 3) `vis` 与 `cur` 仍按 `N=65` 开，和真实节点数不匹配

定义：

```cpp
constexpr int N=65,M=10;
ll ... cur[N];
bitset<N> vis;
```

但节点上界：

```cpp
S=n+n*m+1, T=S+1
```

当 `n=60,m=9` 时，`T=602`。

- `vis[602]` 越界
- `cur[602]` 越界

虽然你把 `dis` 扩成了 `N*M+5`，但 `cur/vis` 仍然不够。

**后果**：RE、随机 WA、死循环都可能出现（未定义行为）。

---

### 4) `dis` 初始化区间写死到常量，不按真实 `T`

代码：

```cpp
fill(dis, dis+1+N*M, 1e18);
```

虽然这次范围比以前大，但这属于“常量拍脑袋”，不是按 `T` 初始化。正确应该是至少到 `T`，即和当前输入规模一致。

**后果**：在你当前约束下可能凑巧够用，但可维护性差，改常量/改建图后容易埋雷。

---

## 三、语义与健壮性问题

### 5) `cur` 只开到 65，DFS 当前弧优化会访问非法内存

DFS：

```cpp
for(ll &i=cur[u]; i<ps[u].size() && ans<flow; ++i)
```

当 `u` 是服务位、S、T 等大编号节点时，`cur[u]` 已经越界引用。

**后果**：即便 SPFA 找到路，DFS 阶段也可能崩溃或写坏内存。

---

### 6) `vis` 复用在 SPFA 和 DFS，且大小不足时风险更大

你在 SPFA 和 DFS 都使用同一个 `vis`。理论上“复用”不是绝对错误，但在当前 `vis` 越界前提下会放大问题，排查困难。

## 四、你这版“看起来改了输出”，但主错误仍在建图阶段

你把最终输出改成了：

```cpp
double rs=ret;
printf("%.2lf",rs/n/1.0);
```

这一点方向是对的（输出费用平均）。

但由于前面边容量/费用写反，以及源汇关键边容量为 0，`ret` 根本不代表正确的最小总等待时间。

---

## 最小修复清单（按优先级）

1. 把三类正向边参数改对：
   - `S -> 车`: `cap=1,cost=0`
   - `车 -> (j,k)`: `cap=1,cost=t[i][j]*k`
   - `(j,k) -> T`: `cap=1,cost=0`
2. `vis/cur/dis/ps` 统一按真实节点上限（至少 `T+1`）开。不要只改 `dis`。
3. SPFA 初始化按 `T`（或 `V`）范围，不用硬编码 `N*M`。
4. 继续输出 `ret/n`，保留两位小数。

---

## 一句话结论

你这版并不是“小 bug”，而是 **建图参数（cap/cost）和容量设定已经破坏了流模型**，再叠加 `cur/vis` 越界，导致结果不可信。先修建图，再修数组范围，最后再谈优化。
